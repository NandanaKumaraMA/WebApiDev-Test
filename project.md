# Taxi Fleet Tracking API — Project Plan

## 1. Business Context

This API powers a **taxi fleet management system** for a company operating in Sri Lanka. Each taxi is fitted with a GPS tracking device that periodically reports the vehicle's location (a "ping"). The API serves three audiences:

- **Operations team** — monitor vehicle locations, view trip histories, manage depots
- **Drivers/mobile app** — report pings, start/end trips
- **External regulators** — query vehicle registrations and compliance data

---

## 2. Assumptions & Clarifications

| # | Assumption | Rationale |
|---|-----------|-----------|
| 1 | A **Station** is a taxi depot/branch office where vehicles are based, not a police station (the existing seed data uses police-station names as sample data). | The business context is a taxi company, not law enforcement. |
| 2 | Each vehicle has exactly **one GPS device** (`device_id`). | The seed data shows a 1:1 vehicle-to-device mapping. |
| 3 | Pings are pushed from the device to the API (not polled). | The `timestamp` field is generated by the device. |
| 4 | A **Trip** starts when a driver accepts a fare and ends when the fare is completed. | Standard taxi business model. |
| 5 | Authentication/authorisation is **out of scope** for this plan. | The existing codebase has no auth middleware. |
| 6 | Data is served from an in-memory seed (no real database). | The current `db.js` loads `seed.json` into memory. |
| 7 | The API is versioned under `/v1/api/`. | Already established in `index.js`. |

**Questions for you:**

- Should Stations be renamed to "Depots" for the taxi context, or keep "Stations"?
- Do you need a Driver entity, or is the driver simply the vehicle's current operator (a field on Vehicle)?
- Should Trips be tracked, or is the ping stream sufficient for your use case?
- Is there a real database planned (PostgreSQL, MySQL), or are you keeping the in-memory seed approach?

## 3. Data Model

### 3.1 Entity-Relationship Overview

```
Province (1) ──< (N) District (1) ──< (N) Station (1) ──< (N) Vehicle
                                                                  │
                                                                  │ (1)
                                                                  │
                                                            (N) Vehicle
                                                                  │
                                                                  │ (N)
                                                                  ▼
                                                            Ping (GPS)
```

#### District
| Field | Type | Description |

| Field | Type | Description |

| Field | Type | Description |

| Field | Type | Description |

## 4. API Routes

### 4.1 Route Map

```
# ── Geographic hierarchy ──────────────────────────────
GET    /v1/api/provinces              # List provinces
GET    /v1/api/provinces/:id          # Get one province
GET    /v1/api/provinces/:id/districts  # Districts in a province

GET    /v1/api/districts              # List districts
GET    /v1/api/districts/:id          # Get one district
GET    /v1/api/districts/:id/stations   # Stations in a district

GET    /v1/api/stations               # List stations
GET    /v1/api/stations/:id           # Get one station
GET    /v1/api/stations/:id/vehicles    # Vehicles at a station

# ── Vehicle tracking ───────────────────────────────────
GET    /v1/api/vehicles               # List vehicles
GET    /v1/api/vehicles/:id           # Get vehicle + last known ping
GET    /v1/api/vehicles/:id/pings     # All pings for a vehicle
GET    /v1/api/vehicles/:id/last-position  # Latest ping only

# ── Pings ─────────────────────────────────────────────
GET    /v1/api/pings                  # List pings (with filters)
GET    /v1/api/pings/:id              # Get one ping

# ── Drivers (new) ──────────────────────────────────────
GET    /v1/api/drivers                # List drivers
GET    /v1/api/drivers/:id            # Get one driver
GET    /v1/api/drivers/:id/trips      # Trips by this driver

# ── Trips (new) ─────────────────────────────────────────
GET    /v1/api/trips                  # List trips
GET    /v1/api/trips/:id              # Get one trip

(NOTE: Write operations (POST/PUT/DELETE) are excluded from v1.
 Add them in v2 when a real database is introduced.)
```

### 4.3 Query Parameters for Collection Endpoints

Standard filters supported on list endpoints:

| Parameter | Type | Applies To | Example |
|-----------|------|-----------|---------|
| `?station_id=` | int | vehicles, drivers | `/v1/api/vehicles?station_id=4` |
| `?district_id=` | int | stations | `/v1/api/stations?district_id=1` |
| `?vehicle_id=` | int | pings, trips | `/v1/api/pings?vehicle_id=1` |
| `?from=` | ISO date | pings, trips | `/v1/api/pings?from=2026-06-14T00:00:00Z` |
| `?to=` | ISO date | pings, trips | `/v1/api/pings?to=2026-06-15T00:00:00Z` |
| `?limit=` | int (1–1000) | all | `/v1/api/pings?limit=50` |
| `?offset=` | int | all | `/v1/api/pings?offset=100` |
| `?status=` | string | vehicles, trips | `/v1/api/vehicles?status=active` |

## 5. Request / Response Representations

### 5.1 Standard Response Envelope

All responses use plain JSON with no wrapper envelope (the resource is returned directly). Error responses follow this shape:

```json
{
  "error": "Vehicle not found"
}
```

## 7. Future Considerations (v2)

- **Write operations** — `POST /vehicles/:id/pings` for devices to push location data, `POST /trips` to start/end trips
- **Authentication** — JWT-based auth for drivers vs. admin roles
- **WebSocket** — Real-time ping streaming for live map view
- **Geofencing** — Alert when a vehicle enters/leaves a defined zone
- **Analytics** — Aggregated endpoints (`/v1/api/analytics/daily-distance`, `/v1/api/analytics/fleet-utilisation`)
- **Pagination metadata** — Add `X-Total-Count` header or `{ data: [...], meta: { total, page, limit } }` envelope
